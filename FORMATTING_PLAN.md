# HTML/CSS/JS 渐进式格式化方案

## 1. 你现在不满意的点

当前效果的问题不是“最终结果错了”，而是“输入过程太丑”：

- `.html` / `.css` / `.js` 如果源内容本身是压成一行的，播放时会先沿着这一行一直打下去。
- 直到全部打完，或者最后再触发一次整篇格式化，文档才变好看。
- 这和真实录屏里的编辑体验不一样，看起来像“机器把一整坨文本灌进去”。

你要的不是“最后格式正确”，而是：

- 输入过程中就像人在写代码。
- 写完一行、一个语句块、一个标签块之后，界面已经接近正常代码形态。
- 至少不能出现“整段全在一行，最后才一下子排版”的效果。

## 2. 当前实现为什么会这样

### 2.1 当前逻辑的本质

当前扩展的主循环本质上还是：

1. 读取源文本
2. 把文本切成字符片段
3. 一个字符一个字符插入
4. 可选地在结束后做一次整篇格式化

这意味着：

- 如果“进入播放前”没有把源文本先变成结构化多行文本，那后面就只能照着一行内容一直打。
- 如果“输入过程中”没有按语言结构触发格式化，那视觉上就只能等最后一次格式化。

### 2.2 现阶段的几个具体短板

#### A. 只有“整篇预格式化”和“结束后格式化”

当前更偏向两种策略：

- 开始前整篇格式化源文档
- 结束后整篇格式化目标文档

缺的是中间态：

- 写完一行后格式化一下
- 写完一个 JS 语句块后格式化一下
- 写完一个 HTML 标签块后整理一下缩进
- 写完一个 CSS rule 后整理一下声明块

#### B. marker 模式没有做语言格式化预处理

现在按 `#marker` 分段播放时，section 文本只是做了换行归一化，没有做语言感知格式化。

结果就是：

- 如果 section 本身是一行压缩代码，播放时就会原样一行打到底。

#### C. CSS 没有内置 fallback formatter

当前兜底格式化只覆盖：

- HTML
- JavaScript
- TypeScript

没有 CSS。  
所以只要 VS Code 环境里没有可用的 CSS formatter，CSS 就几乎一定会保留“一行打到底”的表现。

#### D. `formatOnType` 不是默认主路径

即便已经有 `formatOnType` 能力，它现在也不是默认依赖的核心播放策略。

而你需要的其实是：

- 输入过程里的“渐进式整理”

不是：

- 全部打完之后“补救式整理”

## 3. 目标效果

### 3.1 HTML

输入过程中应该更接近：

```html
<div>
  <h1>Hello</h1>
</div>
```

而不是：

```html
<div><h1>Hello</h1></div>
```

一直到最后才变成多行。

### 3.2 CSS

输入过程中应该更接近：

```css
.card {
  display: flex;
  gap: 12px;
}
```

而不是先出现：

```css
.card{display:flex;gap:12px;}
```

然后最后再整体整理。

### 3.3 JS

输入过程中应该更接近：

```js
function init() {
  const el = document.querySelector(".card");
  console.log(el);
}
```

而不是先打一整行函数，再在收尾阶段重新排版。

## 4. 建议的正确方向

我建议把“播放”从单纯字符流改成“字符流 + 结构触发点”。

### 4.1 保留逐字输入，但增加结构化触发

不是直接改成整行插入。  
还是逐字输入，但要在关键节点做一次整理。

关键节点示例：

- HTML: `>`、`</tag>`、块结束后
- CSS: `;`、`}` 后
- JS: `;`、`{`、`}`、换行后

这样既保留“像人在打字”，又能保证画面不会长时间脏乱。

### 4.2 先把源内容预处理成“适合演示”的文本

不是把用户源文件原样拿来打。  
而是先转成一个“演示版文本”。

规则应该是：

- `.html`：先格式化为多行结构，再逐字播放
- `.css`：先格式化为多行 rule/declaration，再逐字播放
- `.js`：先格式化为多行 statement/block，再逐字播放

这样即便关闭中途格式化，基础观感也不会太差。

### 4.3 marker 模式也必须走同样的格式化链路

这是关键。

不能再是：

- 普通整文件模式有预处理
- marker 模式直接原样播放

应该统一成：

- 无论整文件还是 marker section
- 只要语言是 HTML/CSS/JS
- 都先做同一套“演示前格式化”

### 4.4 CSS 必须补 fallback formatter

这是明确缺口。

如果不补 CSS 兜底，用户环境里一旦没 formatter，CSS 就继续很丑。

最低要求：

- 识别 selector block
- `{` 后换行并增加缩进
- `;` 后换行
- `}` 单独成行并回退缩进

不需要第一版就做成完整 CSS parser，但至少要满足录屏观感。

## 5. 我建议的实现方案

## 方案 A：低风险版本

这个版本最务实，我建议先做这个。

### 核心思路

- 所有 HTML/CSS/JS 源内容先预格式化
- 播放时仍然逐字符输入
- 输入到结构字符后，尝试触发一次轻量格式化
- marker 模式和普通模式统一走同一格式化预处理链路

### 要改的点

1. 为 marker section 增加 `resolveFormattedSourceText`
2. 为 CSS 增加 fallback formatter
3. 扩大 on-type / selection-format 的触发点
4. 把“结束后格式化”降级成兜底，而不是主手段

### 预期效果

- 大部分 `.html` / `.css` / `.js` 看起来会明显更像真人输入
- 改动不算太大
- 风险可控

## 方案 B：更像真实编辑的版本

这个版本效果更好，但复杂度高很多。

### 核心思路

把输入动作拆成更高层的操作：

- 输入 token
- 输入一行
- 回车
- 缩进
- 插入闭合标签
- 格式化当前 selection

这已经不是“单纯逐字符播放”，而是“模拟编辑动作”。

### 优点

- 观感更自然
- 更接近 `code-editing-simulation`

### 缺点

- 实现复杂度明显上升
- HTML/CSS/JS 都要做语言特化
- 测试量会变大

## 6. 我建议你确认的设计结论

如果你同意，我下一步会按下面的结论继续，不再走现在这种“全打一遍最后再格式化”的方向。

### 结论 1

HTML / CSS / JS 都必须先进入“演示前格式化”流程。

### 结论 2

marker 模式不能例外，也必须做格式化预处理。

### 结论 3

CSS 要补一个内置 fallback formatter。

### 结论 4

播放时保留逐字输入，但在结构触发点做增量整理。

### 结论 5

“结束后整篇格式化”只能作为兜底，不能再作为主要观感修复手段。

## 7. 推荐实施顺序

我建议按这个顺序做：

1. 先修格式化链路统一性
2. 再补 CSS fallback formatter
3. 再加播放过程中的结构触发格式化
4. 最后再调观感细节，比如停顿、速度、触发点阈值

这样每一步都能看到明显收益，不会一上来就把复杂度拉满。

## 8. 这份方案对应的最终判断

你的判断是对的。  
当前版本的问题不是“功能不能用”，而是“演示过程不自然”。  
如果目标是做录屏 / 教学 / 展示，这个问题必须优先修，不然观感确实很笨。

我建议下一步直接按“方案 A”落地，不先做更重的“方案 B”。
